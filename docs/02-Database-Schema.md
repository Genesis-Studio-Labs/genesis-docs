## 1. Schema Overview

The Genesis Studio database runs on **Supabase PostgreSQL** and is organized into five logical domains:

| Domain | Description | Managed By |
|---|---|---|
| **Content** | Novels, chapters, genres, tags, languages, banners, release schedules | Directus CMS |
| **User** | User profiles, bookmarks, reading progress, preferences, billing | Supabase (application code) |
| **Commerce** | Wallets (Helix currency), chapter purchases, subscriptions, payments | Supabase (application code) |
| **Engagement** | Singularities, comments, reviews, achievements, badges, quests, events | Mixed (Directus + Supabase) |
| **System** | CMS-managed display content — showcases, featured novels, catalogue spreads, static pages | Directus CMS |

### Management Split

- **Directus CMS tables** — Content editors manage these tables through the Directus admin panel at `edit.genesistudio.com`. Directus controls the schema, migrations, and provides the REST API for reading data. These tables typically have `date_created`, `date_updated`, `user_created`, and `user_updated` audit columns.

- **Supabase-managed tables** — The application code manages these tables directly via the Supabase client SDK. Schema changes are applied through Supabase migrations or the dashboard. These tables are used for user-specific data, commerce transactions, and engagement features that require application-level logic.

> Raw Directus data model exports are available in `misc/directus_data_model/` for reference.

## 2. Conventions

### Naming

- **Table names:** `snake_case`, plural form (e.g., `novels`, `chapters`, `user_profiles`)
- **Column names:** `snake_case` (e.g., `novel_title`, `chapter_number`, `date_created`)
- **Junction tables:** `{singular}_{plural}` or `{table1}_{table2}` (e.g., `novels_genres`, `badges_users`)

### Primary Keys

- Most tables use **UUID** primary keys (generated by PostgreSQL `gen_random_uuid()` or Directus).
- Some tables (particularly Directus-managed lookup tables and junction tables) use **integer auto-increment** primary keys.
- Supabase auth users are referenced via `auth.users.id` (UUID).

### Timestamps

- **Content tables** (Directus-managed): `date_created` (timestamp), `date_updated` (timestamp)
- **Application tables** (Supabase-managed): `created_at` (timestamptz), `updated_at` (timestamptz) — some use `joined_at`, `purchased_at`, `read_at`, etc.

### Status Fields

- Content tables use `status` varchar fields with values: `published`, `draft`, `archived`
- Commerce tables use `status` for transaction states: `pending`, `succeeded`, `failed`, `refunded`, `cancelled`
- Subscription tables use `status` for subscription lifecycle: `active`, `cancelled`, `expired`, `past_due`

- `novels` → `novel_roles` (One-to-Many)

### Database Functions (RPC)

| Function | Parameters | Returns | Purpose |
|---|---|---|---|
| `get_user_sessions(p_user_id uuid)` | User UUID | Sessions table rows | Lists active auth sessions for device management |
| `delete_user_session(p_user_id uuid, p_session_id uuid)` | User UUID + Session UUID | boolean | Revokes a specific auth session |

Both functions use `SECURITY DEFINER` to access the `auth.sessions` table which isn't exposed via PostgREST.
